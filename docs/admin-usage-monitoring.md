# Admin usage monitoring

This document explains the first-launch admin usage monitoring added for Hellowhen Trade.

## Purpose

The usage dashboard is for operational awareness only. It helps the admin understand whether the app is being used, which broad app areas are active, whether the API is slow or erroring, and whether the API/database are basically healthy.

It is not a surveillance feature and must not be used to inspect private user content.

## Admin page

The dashboard is available to admins at:

```txt
/admin/usage
```

The backing API is:

```txt
GET /admin/usage
```

Normal users should not be able to access either the page or the API.

## User heartbeat

Logged-in web users send a lightweight heartbeat to:

```txt
POST /usage/heartbeat
```

The heartbeat stores only operational presence fields:

```txt
userId, when logged in
sessionId, when available
clientId, generated by the web client
appArea
routePattern
lastSeenAt
```

The app area is a coarse label such as:

```txt
trades
trade_detail
proposal_thread
needs
offers
account
admin
```

Route patterns are sanitized and should use placeholders instead of raw private URLs, for example:

```txt
/trades/[tradeId]
/trades/[tradeId]/proposals/[proposalId]
/needs/[needId]
/offers/[offerId]
```

## API request metrics

The API records lightweight timing metrics for normal API requests.

Stored fields:

```txt
userId, when logged in
sessionId, when available
method
safe routePattern
appArea
statusCode
statusGroup
durationMs
createdAt
```

Skipped paths:

```txt
/health
/uploads
/usage/heartbeat
```

This avoids dashboard noise and avoids treating static upload requests as user behavior.

## Usage weight

The dashboard labels recent active users as:

```txt
Lite
Medium
Heavy
```

This is only an operational load estimate. It is based on safe recent signals:

```txt
heartbeat recency
API request count
API error count
average API duration
```

It is not an exact energy, carbon, CPU, or cost calculation.


## Server health snapshot

The admin usage API also returns a lightweight server health snapshot.

It includes:

```txt
API process uptime
API process start time
Node version and runtime environment
API process memory summary
Database ping status
Database ping response time
```

It does not include:

```txt
per-user CPU usage
server-wide CPU graphs
server-wide RAM graphs
disk browsing
process list inspection
shell command output
external monitoring data
```

This is a first-launch health check only. For deeper Lightsail monitoring, use platform monitoring or a dedicated server monitoring tool later.

## Data that must not be tracked

The usage monitoring feature must not store or display:

```txt
private message text
proposal content
support message text
report body text
request bodies
query strings
raw private URLs
uploaded file contents
passwords, tokens, or secrets
exact hidden/private location content
```

When admins need to handle moderation or support, they should use the existing admin moderation/support tools, not the usage dashboard.

## Safe drill-in links

The active users table may include links to:

```txt
/admin/users?userId=<userId>
/users/<userId>
```

These links are intentionally limited to the admin user moderation page and the user's public profile. They must not deep-link to private proposal threads, message bodies, support message bodies, report body text, raw URLs, or other private content from the usage dashboard.

The admin users API can accept a `userId` query parameter so an admin can open a specific account from usage monitoring without broad search. This endpoint remains protected by the existing admin guard.

## Retention

Usage monitoring data is intentionally short-lived:

```txt
presence heartbeat rows: 72 hours
API request metric rows: 72 hours
admin dashboard window: last 15 minutes
live-now window: last 5 minutes
```

Rows older than the retention windows are ignored by dashboard queries and deleted during normal usage writes/reads:

```txt
heartbeat writes
API metric writes, rate-limited to avoid cleanup on every request
admin usage reads
```

This cleanup is best-effort operational cleanup. It is not a legal record-retention system.

## Privacy copy for admins

Use this copy in the admin area:

```txt
This page shows operational presence only. It does not show private messages or proposal content.
```

For the extended API metrics dashboard:

```txt
This page shows operational presence, API timing, simple usage weight, and basic API/database health only. It does not show private messages, proposal content, request bodies, query strings, uploaded content, or raw private URLs.
```

## Production migration

Patch 1 and Patch 2 added Prisma schema migrations. Production should use:

```powershell
npx prisma migrate deploy --schema apps/api/prisma/schema.prisma
```

Do not use `prisma migrate dev` in production.

## Manual QA

1. Log in as a normal user and browse Trades, Needs, Offers, and Account.
2. Confirm `POST /usage/heartbeat` returns `204`.
3. Log in as admin and open `/admin/usage`.
4. Confirm active users appear within 5 minutes.
5. Confirm inactive users disappear from “Live now” after the timeout.
6. Confirm slow route rows use safe patterns like `/trades/[tradeId]`, not raw IDs.
7. Confirm server health shows API uptime, API process memory, and database ping.
8. Confirm no private message, proposal, support, or report text appears.
9. Confirm normal users cannot access `/admin/usage` or `GET /admin/usage`.
10. Confirm old rows are removed after the configured retention window during normal reads/writes.
